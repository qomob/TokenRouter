## 三大路由方式

### 1. 单次路由（Direct Routing）

最简单的策略：每个请求独立评估，直接选择最优模型。

```
请求 → 复杂度评估 → 选择模型 → 执行 → 返回
```

**优点**：延迟最低，实现最简单
**缺点**：评估错误时无纠错机会
**适用**：单次问答、独立任务

### 2. 级联路由（Cascading Routing）

从最便宜的模型开始尝试，质量不达标则逐级升级。

```
请求 → L0模型尝试 → 质量达标? → ✅ 返回
                      ↓ ❌
                 L1模型尝试 → 质量达标? → ✅ 返回
                               ↓ ❌
                          L2模型尝试 → 质量达标? → ✅ 返回
                                        ↓ ❌
                                   L3模型 → 返回
```

**质量判断标准**（根据任务类型选择）：
- 结构化输出：JSON格式是否合法 + 字段是否完整
- 分类任务：置信度是否 > 0.85
- 文本生成：长度是否在预期范围 + 关键信息是否包含
- 代码任务：是否能通过编译/测试

**优点**：成本最优，大部分请求在第一级就解决
**缺点**：延迟增加（最差情况需4次调用）
**适用**：批量处理、非实时场景、成本极度敏感

**级联路由成本测算**：
假设 80% 请求在 L0 解决，15% 在 L1，4% 在 L2，1% 到 L3：
- 平均成本 ≈ 0.8×L0价格 + 0.15×(L0+L1价格) + 0.04×(L0+L1+L2价格) + 0.01×(全部价格)
- 实际节省约 75-85%（因为 L0 解决的请求占绝大多数）

### 3. 混合路由（Hybrid Routing / Agent工作流专用）

在 Agent 工作流的每个步骤独立选择模型。

```
┌─────────────────────────────────────────┐
│ Agent 工作流                              │
│                                          │
│ [意图识别] → L0模型                       │
│      ↓                                   │
│ [参数提取] → L0模型                       │
│      ↓                                   │
│ [知识检索] → 向量数据库（不消耗Token）     │
│      ↓                                   │
│ [核心执行] → L1-L3（按复杂度选择）         │
│      ↓                                   │
│ [质量校验] → L0/L1（按精度要求选择）       │
│      ↓                                   │
│ [输出格式化] → L0模型                     │
└─────────────────────────────────────────┘
```

**关键洞察**：
- Agent 工作流中，80% 的步骤是 L0 级别的"粘合操作"
- 只有核心执行步骤需要强模型
- 一个10步的 Agent 工作流，可能只有1-2步需要 L2+

**混合路由成本对比**（10步Agent工作流）：
- 全旗舰：10步 × L3价格 = 10x
- 混合路由：8步×L0 + 1步×L2 + 1步×L1 ≈ 0.8x
- **节省约 92%**

## 缓存策略

### Provider级缓存（零成本实现）

| 厂商 | 缓存机制 | 节省幅度 | 使用条件 |
|------|---------|---------|---------|
| Anthropic | Prompt Caching | 输入成本降90% | 相同前缀 + 设置 cache_control |
| OpenAI | 自动缓存 | 输入成本降50% | 自动生效，重复前缀 |
| Google | Context Caching | 显著降低 | 手动创建缓存 |

**最佳实践**：
- 系统提示词放最前面，确保每次调用前缀一致
- 工具定义（Tool Schema）放在系统提示词后
- 用户输入放在最后面
- 这样前两部分可以命中缓存，只对最后一部分计费

### 语义缓存（应用层实现）

对语义相似的查询直接返回缓存结果，不调用模型。

```
用户查询 → 生成Embedding → 检索缓存库 → 相似度>阈值? → 返回缓存
                                                   ↓ 否
                                              调用模型 → 存入缓存
```

**适用场景**：FAQ、客服、重复性查询
**预期效果**：约31%的查询可命中缓存，100%成本节省

## Agent工作流路由

### 典型Agent步骤的模型推荐

| Agent步骤 | 任务特征 | 推荐层级 | 推荐模型 |
|-----------|---------|---------|---------|
| 意图识别 | 短输入、分类输出 | L0 | Gemini Flash |
| 参数提取 | 短输入、结构化输出 | L0 | GPT-4o-mini |
| 工具选择 | 短输入、枚举选择 | L0 | Gemini Flash |
| 知识检索 | 向量搜索、无模型调用 | - | 向量数据库 |
| 代码生成 | 中长输入、长输出 | L2 | Claude Sonnet |
| 文本生成 | 中等输入、长输出 | L1-L2 | 按复杂度选择 |
| 结果校验 | 中等输入、短输出 | L0-L1 | 按精度要求选择 |
| 格式化 | 短输入、结构化输出 | L0 | GPT-4o-mini |
| 错误恢复 | 需要分析错误原因 | L1-L2 | 按复杂度选择 |

### 上下文管理策略

Agent 循环中上下文持续增长是最大的成本杀手：

```
第1轮: 1,000 tokens
第5轮: 5,000 tokens
第10轮: 10,000+ tokens（每轮重复计费）
第20轮: 20,000+ tokens
```

**优化策略**：
1. **摘要压缩**：每5轮对历史做一次摘要，将历史压缩到原来的20%
2. **滑动窗口**：只保留最近N轮完整上下文，更早的只保留摘要
3. **按需加载**：不是所有工具结果都需要持续保留在上下文中
4. **分级保留**：L0步骤的详细输出不保留，只保留结论

## 质量评估与自动升级

### 自动升级触发条件

| 信号 | 动作 | 记忆更新 |
|------|------|---------|
| 用户说"这次质量不够" | 同类任务提升1个层级 | 更新该任务类型的最低层级 |
| 用户说"用更好的模型" | 当前任务提升1个层级 | 考虑调整全局策略 |
| 用户说"帮我省钱" | 重新评估是否可以降级 | 策略倾向 cost_first |
| 输出被用户大幅修改 | 同类任务提升1个层级 | 更新该任务类型的推荐层级 |
| 用户连续接受推荐 | 保持当前策略 | 画像稳定性+1 |

### 质量自检模板

对于 L2+ 的任务，可以在输出前做一次轻量自检：

```
自检项（用 L0 模型执行，消耗约100 token）：
1. 输出是否完整回答了问题？
2. 格式是否符合预期？
3. 是否包含明显的错误信息？
```
